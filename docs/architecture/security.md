# SPAL Security Model

---

## Principles

1. SPAL owns league state. External feeds are data sources, not trusted inputs.
2. Never expose the Supabase service role key to the browser.
3. Use Edge Functions for all critical writes. Client cannot bypass validation.
4. Row Level Security protects all exposed Supabase tables.
5. Store minimum personal data.
6. Audit all admin actions.
7. Hash all private tokens before storage.

---

## Roles

| Role | How assigned | Capabilities |
|------|-------------|-------------|
| `admin` | Set directly in `user_roles` by service role | Full read/write across all tables; runs imports; overrides data; manages season and league |
| `manager` | Assigned by admin via invite | Read most data; write own squads before lock; participate in draft |
| `guest` | Presents valid guest token | Read guest-visible data only; no writes |
| `anon` | No login, no token | Read public pages only (`/spal/`, `/spal/laws`, `/spal/history`) |

---

## Row Level Security

RLS is enabled on all exposed tables. The service role bypasses RLS (Edge Functions only).

### General patterns

**Public read (anon + all authenticated):**  
Seasons, leagues (non-private fields), players, matches, standings, fixture groups, Chronicle posts marked `public`.

**Manager read:**  
Own squad, own notifications, Chronicle posts marked `manager` or `public`.

**Manager write (own data only):**  
- `manager_round_squads` â€” insert/update own squads before lock
- `notifications` â€” mark own notifications read

**Admin write (all data):**  
All tables. Admin check: `EXISTS (SELECT 1 FROM user_roles WHERE profile_id = auth.uid() AND role = 'admin')`.

**Guest read (via token):**  
Guest access is validated in the Edge Function `validate-guest-token`. A session or short-lived JWT can be issued for the guest's session to avoid re-validating every request. Guest-visible data excludes email addresses, admin notes, raw payloads, and private audit data.

**No direct write from browser for:**  
- `draft_picks` (must go through `draft-pick` Edge Function)
- `manager_round_squad_players` (must go through `submit-squad` Edge Function)
- `admin_overrides` (must go through `apply-override` Edge Function)
- `import_runs` / `raw_import_payloads` (must go through `run-import` Edge Function)
- `player_match_scores` finalisation (must go through `finalise-round` Edge Function)

## Sensitive data

Never expose to browser or guest views:

| Data | Reason |
|------|--------|
| `auth.users.email` | Personal data |
| Supabase service role key | Full database access |
| External API credentials / tokens | Would expose official feed access |
| Raw import payloads (if containing licensed data) | May contain third-party proprietary data |
| `league_guest_links.token_hash` | The hash, not the raw token â€” raw token is never stored |
| Admin notes in overrides | Internal league administration |
| Audit log details | Admin-only operational data |

---

## Guest links

- Generated by admin via `create-guest-token` Edge Function
- Raw token: cryptographically random, sent to admin once, never stored
- Stored value: `SHA-256(raw_token)` â€” only the hash
- Validation: `validate-guest-token` Edge Function hashes the presented token and compares to stored hash
- Revocable at any time by admin (sets `revoked_at`)
- Optional expiry (`expires_at`)

---

## Invite flow

1. Admin calls `send-invite` Edge Function with manager's email
2. Edge Function sends invite email via email provider
3. Invite contains a Supabase Auth magic link or invite token
4. Manager clicks link, completes sign-up
5. Edge Function assigns `manager` role in `user_roles`

Invite tokens are single-use and expire. No self-registration outside the invite flow.

---

## Critical server-side operations

The following must always run through Edge Functions. They must never be callable directly by a browser client with the anon key:

- `draft-pick` â€” validate and record a draft pick
- `submit-squad` â€” validate and record a squad submission
- `run-import` â€” execute a data source import
- `calculate-scores` â€” recalculate manager and round scores
- `finalise-round` â€” lock scores and update standings
- `apply-override` â€” apply and audit an admin override
- `send-invite` â€” send manager invite email
- `create-guest-token` â€” generate and store hashed guest token
- `validate-guest-token` â€” check guest token validity and issue session

Each Edge Function performs full server-side validation before writing. Client-side validation improves UX but is never the only gate.

## No scraping or licensed data exposure

SPAL must not:
- Expose official feed API tokens or credentials to the browser
- Re-publish official fantasy scores or data in a way that creates legal risk
- Scrape private or licensed data without appropriate rights

All official data is:
- Imported server-side only
- Stored as raw payload for audit
- Normalised into canonical SPAL records
- Overridable by admin if source data is incorrect
