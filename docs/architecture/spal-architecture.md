# SPAL Architecture Overview

---

## Hosting and routing

SPAL lives at `nosky.co.uk/spal/` as a sub-application of the existing Netlify site.

```
nosky.co.uk/           — existing plain HTML/CSS/JS site (index.html, style.css, etc.)
nosky.co.uk/spal/      — SPAL React SPA entry point
nosky.co.uk/spal/*     — all SPAL routes handled by React Router (client-side)
```

The existing nosky.co.uk site continues to operate unchanged. SPAL is a separate Vite app that lives in a `/spal` subdirectory of the repo. Its build output is placed into the Netlify publish directory alongside the static site files.

### Repository structure

```
my-website/
  index.html          — nosky.co.uk homepage (plain HTML)
  style.css           — shared nosky styles
  fpl.html            — FPL tool (plain HTML)
  cyber.html          — Cyber security page (plain HTML)
  nav.js              — shared nav dropdown script
  netlify/
    functions/        — Netlify Functions (fpl.js, fpl-team.js, future Edge Functions)
  spal-app/           — SPAL Vite/React source (versioned)
    src/
    public/
    index.html        — Vite entry point template
    vite.config.ts    — base: '/spal/', outDir: '../spal', emptyOutDir: true
    package.json
    tailwind.config.ts
    tsconfig.json
  spal/               — GENERATED — Vite build output (gitignored, served at /spal/)
  netlify.toml        — build config and redirect rules
  docs/
  .gitignore          — includes /spal/
```

**Source lives in `spal-app/`.** It is a self-contained Node project: own `package.json`, own `tsconfig.json`, own Tailwind config. Nothing in `spal-app/` is served directly.

**Output goes to `spal/` at the repo root.** Vite is configured with `build.outDir: '../spal'` and `build.emptyOutDir: true`. The `spal/` directory is entirely generated — it is gitignored and contains only build artifacts. `emptyOutDir: true` is required because Vite refuses to empty a directory outside the project root without explicit opt-in.

**Netlify serves from the repo root (`publish = "."`)** which is unchanged. Since `spal/` exists at the repo root after the build, all assets are served at `/spal/` — exactly as required. The static site files (`index.html`, `style.css`, etc.) are untouched by the Vite build.

### Netlify configuration

```toml
[build]
  publish = "."
  command = "cd spal-app && npm ci && npm run build"

[functions]
  directory = "netlify/functions"

[[redirects]]
  # Deep-link support for React Router — any /spal/* path that doesn't
  # match a real file falls through to the SPA shell
  from = "/spal/*"
  to   = "/spal/index.html"
  status = 200
```

No redirect is needed for paths outside `/spal/` — Netlify serves them as static files from the repo root.

The `[functions]` block is explicit even though `netlify/functions` is the default, to make the configuration unambiguous as the project grows.

### `.gitignore` addition

```
# SPAL build output — generated by Vite, not source code
/spal/
```

The leading `/` anchors to the repo root, so a hypothetical nested `spal/` directory elsewhere would not be accidentally ignored.

### CSS isolation

The nosky.co.uk static pages load `style.css` explicitly via `<link>`. The SPAL React app is bundled by Vite and loads its own CSS via `<link>` tags injected into `/spal/index.html`. The two stylesheets do not share a load path — they are isolated by construction.

SPAL's `/spal/index.html` must include the nosky site header and footer (or a matching static shell) so the visual transition between the static site and the SPAL app is seamless. The simplest approach is to copy the header/footer HTML into `/spal/index.html` and keep it in sync manually — or render them as React components that replicate the nosky header exactly.

## Front end

```
Vite
React 18+
TypeScript
Tailwind CSS
shadcn/ui (component library)
React Router (client-side routing within /spal)
Supabase JS client (browser-safe operations only)
```

### What runs in the browser

- All UI rendering and navigation
- Read operations via Supabase JS client (subject to RLS)
- Optimistic UI for squad builder
- Realtime draft board via Supabase Realtime channel
- Form validation (client-side, for UX only — server always re-validates)

### What does NOT run in the browser

- Import execution
- Scoring recalculation
- Draft pick creation (validation and write)
- Squad submission (validation and write)
- Admin overrides
- Email sending
- Guest token creation and validation
- Any operation requiring service role key

---

## Back end

```
Supabase Auth        — user accounts, invite flow, JWTs
Supabase Postgres    — all persistent data
Supabase RLS         — row-level access control
Supabase Realtime    — live draft board updates
Supabase Edge Functions — server-side critical operations
Supabase Storage     — uploaded files (CSV imports, legacy spreadsheet, screenshots)
```

### Edge Functions

Edge Functions handle all server-side operations where correctness, security, or access to service role key is required:

| Function | Responsibility |
|----------|---------------|
| `draft-pick` | Validate and record a draft pick |
| `submit-squad` | Validate and record a round squad |
| `run-import` | Execute a data source import |
| `calculate-scores` | Recalculate manager and round scores |
| `finalise-round` | Lock scores and update standings |
| `apply-override` | Apply and audit an admin override |
| `send-invite` | Send manager invite email |
| `create-guest-token` | Generate and store hashed guest token |
| `validate-guest-token` | Check guest token validity |

## Data flow

### Reads (browser-safe)

```
Browser → Supabase JS client → Supabase Postgres (RLS enforced)
```

All read queries go directly from the browser through the Supabase JS client. RLS ensures each user can only see what their role permits.

### Critical writes

```
Browser → Edge Function (validates, uses service role) → Supabase Postgres
```

Draft picks, squad submissions, imports, scoring, and admin overrides never write directly from the browser. They go through Edge Functions which perform server-side validation before writing.

### Realtime (draft room)

```
Browser → Supabase Realtime channel → all connected clients
```

The draft room subscribes to a Realtime channel. When a pick is made (via Edge Function), the updated draft state is broadcast to all connected browsers without polling.

## External data sources

```
Official fantasy feed / scraped data
        ↓
officialFantasyAdapter (normalises to CanonicalPlayer, CanonicalFixture, etc.)
        ↓
raw_import_payloads (stored unchanged for audit)
        ↓
import normalisation + admin review
        ↓
canonical tables (players, matches, player_match_scores, etc.)
```

No module reads directly from an external API. All external data passes through adapters.

```

  modules/

    league/         — dashboard, manager list, league overview
    draft/          — draft room, draft board, draft admin

    scoring/        — score display, round results

    admin/          — season setup, imports, overrides, penalties
    imports/        — adapter management, import health
    notifications/  — in-app notifications
  shared/
    components/     — reusable UI components
    hooks/          — shared React hooks
    lib/            — Supabase client, utilities
    types/          — TypeScript types
```

## Environments

| Environment | URL | Notes |
|-------------|-----|-------|
| Production | `nosky.co.uk/spal/` | Netlify production deploy |
| Preview | Netlify deploy preview per PR | Automatic per pull request |
| Local | `localhost:5173/spal/` | Vite dev server |

Environment variables (never committed):
- `VITE_SUPABASE_URL` — Supabase project URL (safe to expose to browser)
- `VITE_SUPABASE_ANON_KEY` — Supabase anon key (safe to expose; RLS is the security layer)
- `SUPABASE_SERVICE_ROLE_KEY` — service role key (Edge Functions only, never browser)
